# DECISION P3.17 — Risk / Trade Management Engine detail set (D17, §7 pins)

**Date:** 2026-07-18 · **Status:** Accepted (engine-wise directive; pins per
the D9–D16 precedent) · **Scope:** implementation decision only — zero
architecture change. Sole specification for the Risk / Trade Management
Engine (§7 PLANNING + SUGGESTED MANAGEMENT; roadmap P3.17).

---

## D17.1 — Scope ruling

The engine = the §7 **planning math** (risk sizing, suggested qty,
fee-adjusted net RR, RR-floor validation, deterministic rejection
reasons) plus the §7 **SUGGESTED MANAGEMENT display-only guidance text**
(v1.2: management guidance is text, never automation). It is a **pure
capability module with no composition wiring** — its inputs (direction,
entry, SL, TP) exist only once S1–S3 strategies land (P3.12–P3.16), the
exact precedent of P1.1 (ATR built unwired) and R1 (persistence as
capability). It NEVER generates setups, market structure, or executions.

Explicitly out of scope (owners recorded): recommendation lifecycle —
invalidation / expiry / outcome eval (P4.2+, gated on the owner's P4.1
[DECISION] A16/A17); persistence of plans (`recommendations` rows are
P3.18's); journal / daily-loss / psychology inputs (P4.8–P4.9, G5);
equity or account state from any exchange (v1.2: no broker auth — equity
is a caller-supplied display input); Kelly / adaptive / volatility
sizing (not in the roadmap — §7 is fixed-fraction).

**G6 wiring deferral (recorded):** D16.2 left the qualification G6 gate
as a flagged pass "until the planner (P3.17)". The QualificationEngine
is FROZEN and its update signature takes no plan input, so connecting G6
to real RR values requires a frozen-interface extension — an owner
decision at strategy-integration time. This engine provides the RR
computation and the `rr_floor_ok` flag; G6 itself stays flagged.

## D17.2 — Planning math (§7 verbatim; boundaries and mirror pinned)

Frozen literals (module constants, not config): `RISK_PCT = 0.005`
(§7 "risk_amt = equity × 0.5%"; the 0.25–0.5% band's formula value),
`NET_RR_TP1_FLOOR = 1.0` (§7 "reject if net RR(TP1) < 1.0" — rejection
is **strict <**, so exactly 1.0 passes), `RR_TP2_FLOOR = 1.5` (§6 G6
"to TP2 ≥ 1.5" — **inclusive**), `DEFAULT_TAKER_FEE = 0.0005`
(§7 "Delta ~0.05% taker/side").

Given a valid plan request (direction, entry, sl, tp1, optional tp2,
equity, taker_fee):

- `risk_amt = equity × RISK_PCT`
- `r_per_unit = |entry − sl|`
- `qty = risk_amt / r_per_unit` — **SUGGESTED, display only** (§7)
- `fee_per_unit = entry × taker_fee × 2` (from §7's
  `fees = qty × entry × taker_fee × 2`, divided by qty)
- net RR, LONG (§7 verbatim):
  `(tp − entry − fee_per_unit) / (entry − sl + fee_per_unit)`;
  SHORT is the pinned mirror:
  `(entry − tp − fee_per_unit) / (sl − entry + fee_per_unit)`
- Rejection: `net_rr_tp1 < 1.0` (strict) → status `rejected`, reason
  recorded; otherwise status `suggested`.
- `rr_floor_ok` (the G6-alignment flag, informational until G6 wiring):
  `net_rr_tp1 ≥ 1.0 AND (tp2 absent OR net_rr_tp2 ≥ 1.5)` — inclusive
  per §6.

## D17.3 — Validation set and rejection reasons (pinned, minimal)

Geometry checks run first; any failure → status `rejected` with every
failing reason (deterministic pinned strings, evaluation order fixed),
and all derived numbers `None` (D7: refuse-until-valid — no math on
false inputs):

1. `direction` must be `LONG` or `SHORT` → "unknown direction".
2. `equity > 0` → "equity must be positive".
3. `taker_fee ≥ 0` → "taker fee must be non-negative".
4. Prices `> 0` (entry, sl, tp1, tp2 if given) → "prices must be
   positive".
5. SL strictly on the loss side (LONG `sl < entry`, SHORT `sl > entry`;
   equality fails — zero-distance risk is undefined) → "stop loss not on
   the loss side".
6. TP1 strictly on the profit side (LONG `tp1 > entry`, SHORT
   `tp1 < entry`) → "tp1 not on the profit side".
7. TP2, when given, strictly beyond TP1 (LONG `tp2 > tp1`, SHORT
   `tp2 < tp1`) → "tp2 not beyond tp1".

The net-RR rejection reason: "net RR to TP1 below 1.0 after fees".

## D17.4 — Suggested management guidance (§7, display-only text)

`management_guidance(plan)` → deterministic tuple of display lines for a
`suggested` plan (empty for `rejected`), the §7 items verbatim with the
plan's computed levels substituted: SL→BE at +1R (BE = entry, +1R level
= entry ± r_per_unit), 50% off at TP1 + trail at last confirmed 1m
swing, 15-minute time stop, exit on an opposite displacement candle
(> 1.5×ATR — the §7 literal, quoted in text only; this engine computes
no ATR). Pure text; the platform never acts on it (v1.2).

## D17.5 — Composition, payload, persistence, harness

None, by design: no composition wiring (D17.1), payload unchanged,
frontend untouched, no persistence (P3.18), and no determinism-harness
extension (the harness hashes the payload; nothing new is emitted —
"extend only where required"). Determinism is covered at the unit level:
the module is pure (`plan_trade` and `management_guidance` are
functions of their arguments alone — no clock, no randomness, no state).

## Freeze-audit record (2026-07-18)

**One genuine defect found (adversarial auditor, machine-verified),
fixed + regression-tested before freeze:** the guidance text formatted
levels with 6-significant-digit `%g`, which corrupts BTCUSDT-scale
prices — 118456.7 and 118456.9 both rendered "118457", collapsing BE and
+1R into the same displayed number, and levels ≥ 1e6 went scientific.
Fixed to `:.10g` (sub-tick exact for v1 symbols, no scientific below
1e10); the displayed-levels regression test pins the collision case.
Also fixed at audit: the word "body" removed from the displacement-exit
line (§7's literal has no measurement qualifier) and the TradePlan
docstring aligned with the two rejection modes.

**Behavior-identical readings, recorded:**

- **Reason gating:** checks 5–7 presuppose checks 1 and 4 passing (side
  geometry against a non-positive or direction-less input is not
  meaningful — D7). On doubly-invalid corner inputs the reasons tuple
  omits side faults that are individually evaluable; status and the
  all-None numerics are identical either way. "Every failing reason"
  reads within that gating.
- **Non-finite inputs:** the D17.3 validation set pins sign checks, not
  finiteness; inf/NaN garbage-in yields garbage plans (e.g.
  equity=inf → qty=inf suggested). Unreachable from roadmap callers
  (P3.13+ supply finite exchange prices); recorded, not guarded.
- **Net-RR rejection keeps its computed numbers** (they are the
  rejection's evidence); only geometry failures null the derived fields
  — D17.2/D17.3 scoped exactly this way.
