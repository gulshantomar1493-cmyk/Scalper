## D9 — Regime constants as config; momentum-shift threshold; replay distribution logging

### Constants (defaults = the frozen formulas' own literals; nothing optimized)

| Config key | Default | Source |
|---|---|---|
| `regime.compression_ratio` | `0.6` | §4.2 verbatim: coil when `ATR(14,1m) < 0.6 × ATR(14,5m)` |
| `regime.expansion_ratio` | `1.5` | §4.2 verbatim: expansion when `ATR(14,1m) > 1.5 × rolling median ATR` |
| `regime.median_window_bars` | `240` | The one value §4.2 left unspecified; adopted as proposed by the roadmap's A7 row |
| `momentum.shift_accel_atr_ratio` | `0.1` | §4.7 names a threshold without units; pinned as `ratio × ATR(14,1m)` (ATR-scaled per §12's "ATR-scaled thresholds everywhere"); `0.1` is an explicitly **arbitrary, uncalibrated** starting constant owned by P5.3 |

All four are static per process, read once at startup through the existing config chain (`config.example.yaml` → `config.yaml` → env vars). One constant set for both symbols in v1.

### Pinned definitions (P1.4/P1.2 make zero further choices)

- **Inputs, per symbol:** `ATR(14)` on 1m and 5m (the P1.1 instances) and the rolling **median** of the last `median_window_bars` values of ATR(14,1m), over closed 1m candles only.
- **Precedence:** evaluate **expansion first**, then coil, else normal (`expansion > coil > normal`). An expanding market invalidates coil-based assumptions, so it wins the (rare but possible) tie.
- **Warm-up:** regime is `None` ("unknown") until ATR 1m, ATR 5m and a full median window are all warm. Unknown is a first-class output; consumers treat it as metric-inactive (the A3 refuse-until-ready philosophy).
- **Momentum shift (§4.7):** `velocity sign flip AND |acceleration| > shift_accel_atr_ratio × ATR(14,1m)`; `False` whenever any input is not warm.

### Replay distribution logging (observation only, minimal)

Per replay run and per symbol the classifier keeps four plain counters — bars classified `coil` / `normal` / `expansion` / `unknown` — and on replay completion emits **one INFO summary line per symbol**: counts, percentages, the run's date range, and the constants in effect. Per-bar classification is DEBUG-only. No persistence, no tables, no files, no analytics, no dashboards. This data *informs* P5.3's calibration later; **nothing in P1–P4 may adjust constants from it.**

### Non-goals (frozen)

No calibration against historical data in this phase · no adaptive/dynamic/self-tuning thresholds, ever, without a new owner decision · no per-symbol constant overrides in v1 · no new events, tables, or config machinery beyond these four keys in the existing chain.

**Why.** A7 flagged the compression constant as likely mis-calibrated for 1m (1m ATR is typically already below 0.6× the 5m ATR); hard-coding guesses would bake the suspicion in. Config + logged distributions keep the frozen formulas intact while making P5.3's evidence-based calibration possible.

**Consumers.** P1.2 (`momentum.shift_accel_atr_ratio`), P1.4 (regime keys + precedence + warm-up + counters), P5.3 (sweeps these keys against the holdout discipline).

## Consistency check

Formulas unchanged from §4.2/§4.7 — only literals factored out and two previously-unspecified details (median window, threshold units) pinned, exactly as the roadmap's A7 row instructed. Static-per-process config matches the P0.2 chain; refuse-until-warm matches A3 and the P1.1 ATR contract; ATR-scaling matches §12. No conflict with D1–D8; no roadmap or blueprint edits.
