# DECISION P0.4 — engine_version scheme + ops choices (A19, A20)

**Date:** 2026-07-14 · **Status:** Accepted · **Scope:** implementation decisions only — zero architecture change.
Everything below fills in details that Architecture v1.2 already calls for. Nothing new is designed.

---

## D1 — engine_version scheme (A19)

**Decision.** `signals.engine_version` (column already in Architecture §3) is a compact text stamp:

```
<git-short-hash>+<engine>=<n>[;<engine>=<n>...]     e.g.  a1b2c3d+structure=1;trendline=1;liquidity=1;smc=1;volume=1;strategy=1
```

Each engine module declares one integer constant `ENGINE_VERSION = 1`, bumped by hand **whenever that engine's logic or thresholds change**. The stamp is assembled at startup (git hash read once via `git rev-parse --short HEAD`; falls back to `nogit` outside a repo) and written on every signal row.

**Why.** Forensics and the determinism gate need to know exactly which logic produced a signal. Two replay runs are only comparable when their stamps match; a stats analysis (P5) must never silently mix engine generations.

## D2 — Candle partition automation (A20)

**Decision.** No extensions, no external schedulers. Migration 001 (P0.5) ships one idempotent SQL function `ensure_candle_partitions()` that creates the current and next month's partitions (`CREATE TABLE IF NOT EXISTS … PARTITION OF candles FOR VALUES FROM … TO …`). The app calls it at startup and after each UTC midnight rollover — inside the existing single process. pg_partman and cron are explicitly not used.

**Why.** Architecture §3 mandates monthly range partitions; someone must create them ahead of the data. A self-contained SQL function keeps the frozen "single process, minimal infra" rule — no new moving parts, nothing to install on the server, and replay bootstraps (which insert historical months) can call the same function per month.

## D3 — API auth approach (A20)

**Decision.** Single-user static bearer token. Token value lives only in the environment/local config (`MARKETSCALPER_API_TOKEN` — config layer 3, never in git, per the P0.2 no-secrets rule). REST: `Authorization: Bearer <token>` checked by one FastAPI dependency. WebSocket: same token presented at handshake (query param or header); connection refused otherwise. No accounts, no sessions, no OAuth, no roles.

**Why.** Roadmap P0.21 requires "single-user token auth" — this is the smallest mechanism that satisfies it for a solo owner on a self-hosted server. Anything richer is over-engineering (forbidden by CLAUDE.md rule 4) for a platform that, by frozen v1.2 scope, holds no order-execution capability — the token protects market analysis and journal data only.

## D4 — systemd plan (A20)

**Decision.** One unit file `deployment/marketscalper.service` (written at P0.27), provider-neutral:
`Type=simple`, `ExecStart=<venv>/bin/marketscalper`, `Restart=always` + `RestartSec=5`, `WantedBy=multi-user.target`, dedicated non-root user, `EnvironmentFile=/etc/marketscalper/env` holding the secrets (DSN, API token). stdout/stderr go to journald; the app's own rotating file log (P0.2) stays authoritative for analysis. One service = one process = the whole platform, per Architecture §2.

**Why.** The frozen deployment row requires a persistent single process (WS connections; request-scoped platforms explicitly rejected) surviving crashes and reboots; `Restart=always` + journald is the standard, smallest way to get that on any Linux server, naming no hosting provider.
