# DECISION P4.9 — Psychology guards + the G5 seam (D23)

**Date:** 2026-07-19 · **Status:** Accepted (autonomous execution
directive; the D16.2 pre-recorded resolution of the G5 placeholder —
"G5 until the journal, P4") · **Scope:** implementation decision only —
zero architecture change. Sole specification for roadmap P4.9.

§8's psychology guards are rule-based (not judgmental). They feed G5 (the
§6 risk-budget hard gate), which is a flagged placeholder pass today
(D16.2). This decision makes G5 real when psychology state is available.

## D23.1 — The guard (pure, in-memory, live-only)

`engines/psychology.py`: `PsychologyGuard` — a composition-owned,
account-level (shared across symbols) in-memory tracker of the owner's
TAKEN trades. It holds `(logged_at, symbol, result)` records, pruned to
the current UTC day plus the revenge window. Two methods:

- `record_taken(logged_at, symbol, result)` — called when the owner logs
  a journal row `taken=True` with a `result` (win/loss/be), from the P4.8
  PATCH endpoint. Skipped trades are never recorded.
- `evaluate(now, symbol) → G5State` — the per-bar G5 verdict.

It is LIVE-ONLY: manual journal data exists only in a forward-run.
Replay/tests construct no guard → G5 stays the D16.2 flagged pass (legacy
behavior byte-identical). The guard is therefore OUTSIDE the deterministic
replay contract (§0 rule 2 is unaffected — it never runs in replay).

## D23.2 — The rules (§8 verbatim; thresholds uncalibrated, P5-owned)

Over the owner's taken trades in the current UTC day (`taken_today` =
count of records whose `logged_at` UTC date == `now` UTC date):

- **revenge** — a new signal within 5 minutes after a logged LOSS on the
  SAME symbol (`0 ≤ now − logged_at < 5 min`, `result == 'loss'`) → G5
  **FAIL** (`REVENGE_WINDOW_MIN = 5`).
- **hard lock** — `taken_today > 8` → G5 **FAIL** ("locked") until the
  next UTC day (`OVERTRADE_LOCK = 8`).
- **overtrade warn** — `taken_today > 6` (and not locked) → **WARN**: G5
  still PASSES, the warning is surfaced in the gate detail
  (`OVERTRADE_WARN = 6`). A warning is not a block (§8: "warn").

`G5State`: `passed` (bool), `locked` (bool), `warn` (bool),
`detail` (the gate detail string). `passed = not (revenge or locked)`.

## D23.3 — Time model (accepted approximation)

The signal side uses the candle `ts` (market time — in a live forward-run
the feed is real-time so `candle.ts ≈ wall-clock`), the log side uses
`datetime.now(UTC)` at PATCH time. The sub-second-to-seconds skew between
market time and wall-clock in a live feed is negligible against a
5-minute revenge window and a per-day count — recorded as an accepted
approximation. Both are UTC (the platform's discipline). The guard is
live-only, so this never affects replay determinism.

## D23.4 — The G5 seam (the D16.2 pre-recorded resolution)

`QualificationEngine.update(..., psych=None)` gains one optional keyword
(the D21.3 volume-seam / G1-clock / G2-spread precedent). `psych=None`
→ G5 is the unchanged flagged placeholder pass (legacy, byte-identical —
every existing test and the V1/V2 determinism streams are unaffected).
`psych` = a `G5State` → G5 = `GateResult("G5", psych.passed, False,
psych.detail)` (a real, non-flagged gate). A G5 FAIL makes the whole bar
`NO_SIGNAL` (§6: any gate fail → no signal), so during a revenge window
or a hard lock the platform recommends nothing — the intended
behavioral circuit-breaker. This is the minimal sanctioned change to the
frozen QualificationEngine, pre-recorded at D16.2.

## D23.5 — Composition + API wiring

- `main()` (live) builds one `PsychologyGuard`, passes it to
  `_wire_structure_engines` (each pipeline calls
  `guard.evaluate(candle.ts, symbol)` and threads the result to
  `qual.update(psych=...)`) AND to `create_app` (the PATCH endpoint feeds
  it). Replay/tests pass no guard → `psych=None` per bar.
- The P4.8 PATCH `/journal/{id}` endpoint, when the merged row is
  `taken=True` with a non-null `result`, resolves the recommendation's
  symbol (via its signal row) and calls
  `guard.record_taken(datetime.now(UTC), symbol, result)`. Re-logging the
  same journal row does not double-count (the guard keys taken records by
  `recommendation_id`, updating in place).

## D23.6 — Scope boundaries

- Only TAKEN trades count (Skipped never recorded). Overtrade counts all
  taken trades; revenge counts taken LOSSES on the same symbol.
- The "violation log" guard (§8: manual deviation from guidance → journal
  tag) is a self-report journal tag — a UI/journal concern (P4.7/P4.12),
  not a G5 input; not built here.
- The guard is not persisted (in-memory, rebuilt on restart from the
  day's journal — a restart-rehydration hook is left to P4.13 ops if
  needed; recorded, out of P4.9 scope). A restart mid-day resets the
  count to 0 until trades are re-logged — accepted (the guard is a
  best-effort behavioral aid, not a hard accounting control).
- **Bounded state**: `record_taken` self-prunes (drops records older than
  the current UTC day minus the revenge window) — records only ENTER via
  that method, so the map stays ~current-day-sized on a long forward-run
  (`evaluate` already filters stale records out of its output, so pruning
  is a memory concern only, never a correctness one).

## D23.7 — engine_version (D1)

`qualification.ENGINE_VERSION` stays **1**, consistent with the P3.18
volume-seam precedent: with `psych=None` the G5 branch is byte-identical
to the pre-P4.9 flagged placeholder, and the D1 stamp's
`<git-short-hash>` prefix already distinguishes the P3.18/P4.9 live eras
per commit. The per-engine integer marks a change to the engine's
INHERENT logic; a pre-recorded flagged-placeholder resolution (D16.2)
that is inert without its composition-provided input is not that. Live
signals always carry the real G5 (the guard is always wired in `main()`);
replay persists nothing.

`psychology.ENGINE_VERSION` (and the P4.2/P4.3 `lifecycle`/`evaluator`
constants) were added to the recorder's `_STAMP_ENGINES` roster so every
declared version is forensically live on the `signals.engine_version`
stamp — the psychology guard shapes a recorded signal's G5/verdict, so
D1 ("know which logic produced a signal") applies. No dead constants.
